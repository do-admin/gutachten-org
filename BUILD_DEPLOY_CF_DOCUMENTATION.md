# Cloudflare Build and Deploy Script Documentation

## Overview

The `build-deploy-cf` script (`src/scripts/build-deploy-cf.mjs`) is a comprehensive deployment tool for building and deploying static sites to Cloudflare R2 storage with automatic routing configuration. It handles the complete pipeline from static site generation to deployment and routing setup.

## Performance Optimization: rclone Integration

The R2 deployer now uses **rclone** for uploads, providing **5-20x faster** deployment speeds. The system automatically detects if rclone is installed and uses it when available, falling back to the custom S3 SDK implementation if rclone is not found.

### Installing rclone (Recommended)

For fastest deployments, install rclone:

- **macOS**: `brew install rclone`
- **Linux**: `curl https://rclone.org/install.sh | sudo bash`
- **Windows**: Download from [https://rclone.org/downloads/](https://rclone.org/downloads/)

The deployment script will automatically use rclone if available, or fall back to the custom uploader if not installed.

## What It Does

The script performs the following operations:

1. **Generates sitemaps** for SEO optimization
2. **Generates llm.txt files** (optional) for AI/LLM indexing
3. **Builds static sites** using Next.js static export
   - Automatically disables text editor and SEO buttons (`ENABLE_TEXT_EDITOR=false`)
   - Supports external API configuration for email and reCAPTCHA
4. **Deploys to Cloudflare R2** storage bucket
   - Uses rclone for fast incremental uploads (if available)
   - Falls back to custom S3 SDK if rclone not installed
   - Organizes files by branch: `sites/{siteId}/{branch}/`
5. **Configures KV routes** for domain-to-R2-path mapping
   - Maps hostname to R2 path prefix
   - Stores deployment metadata (siteId, branch, updatedAt)
6. **Sets up Worker routes** for custom domains and wildcard subdomains
   - Ensures wildcard route `*.hero-pages.com/*`
   - Adds custom domain routes (final deployments only)
7. **Creates DNS records** for hero-pages.com subdomains
   - Creates CNAME records (proxied)
   - Adds custom domains to Worker for SSL provisioning
   - Waits for SSL certificate provisioning
8. **Creates www redirects** for apex domains (final deployments only)
   - Redirects `www.{domain}` â†’ `{domain}` (301, preserves path and query)
9. **Creates trailing slash normalization** (final deployments only)
   - Adds trailing slashes to paths (ignores files)

### Deployment Flow

1. **Pre-build Phase**:
   - Validates Cloudflare configuration
   - Generates sitemaps for selected sites
   - Optionally generates llm.txt files

2. **Build Phase**:
   - Runs `generate-static.mjs` to create static exports
   - Outputs to `static-builds/{productId}/` directory

3. **Deploy Phase**:
   - Uploads static files to R2 bucket at `sites/{siteId}/{branch}/`
   - Sets up KV route mapping: `hostname â†’ R2 path`
   - Configures Worker routes for custom domains
   - Creates www redirects (final deployments only)

## Usage

### Interactive Mode

Run without arguments to enter interactive mode:

```bash
pnpm build:deploy:cf
```

The script will prompt you for:

1. **Deployment type**: Test or Final (always prompted, even in command-line mode)
2. **Mode**: Generate, Deploy, Both, or Post-deploy
3. **llm.txt generation**: Yes/No (with double confirmation if yes)
4. **Site selection**: All sites or specific sites

### Command-Line Mode

```bash
pnpm build:deploy:cf [options] [product-ids...]
```

## Command-Line Options

| Option               | Short | Description                                      |
| -------------------- | ----- | ------------------------------------------------ |
| `--generate`         | `-g`  | Generate static builds only (no deployment)      |
| `--deploy`           | `-d`  | Deploy existing builds only (no generation)      |
| `--both`             | `-b`  | Generate and deploy (default mode)               |
| `--post-deploy`      | `-p`  | Post-deployment setup only (routes & redirects)  |
| `--all`              | `-a`  | Process all products from config                 |
| `--generate-llm-txt` | `-l`  | Generate llm.txt files (requires OPENAI_API_KEY) |
| `--help`             | `-h`  | Show help message                                |

### Option Details

#### `--generate` / `-g`

- **What it does**: Only generates static builds
- **When to use**: When you want to build sites without deploying
- **Output**: Creates static builds in `static-builds/{productId}/`
- **Note**: Deployment is skipped

#### `--deploy` / `-d`

- **What it does**: Only deploys existing static builds
- **When to use**: When builds already exist and you only want to deploy
- **Requires**: Pre-existing builds in `static-builds/{productId}/`
- **Note**: Generation is skipped

#### `--both` / `-b`

- **What it does**: Generates static builds AND deploys them
- **When to use**: Full pipeline execution (default)
- **Output**: Builds sites and deploys to R2

#### `--post-deploy` / `-p`

- **What it does**: Sets up routes and redirects only (no R2 deployment)
- **When to use**: When you need to reconfigure routing without redeploying files
- **Requires**: Existing deployment in R2 (uses R2 path from KV or constructs it)
- **Sets up**: KV routes, Worker routes, DNS records, www redirects, trailing slash rules
- **Note**: Useful for fixing routing issues or updating redirect rules

**Example**:

```bash
pnpm build:deploy:cf --post-deploy bayern
# Interactive: Select deployment type
# Sets up routes and redirects without uploading files
```

#### `--all` / `-a`

- **What it does**: Processes all sites from `multi-page-config.json`
- **When to use**: When you want to deploy all configured sites
- **Note**: Can be combined with other options

#### `--generate-llm-txt` / `-l`

- **What it does**: Generates llm.txt files for AI/LLM indexing
- **When to use**: When content is finalized and ready for review
- **Requires**: `OPENAI_API_KEY` environment variable
- **Warning**: Incurs API costs, use carefully
- **Note**: Should only be run when content is finalized

## Deployment Types

The script supports two deployment types:

### ğŸ§ª Test Deployment

- **Branch**: `test`
- **Access**: Only via `{productId}.hero-pages.com`
- **Use case**: Preview/testing before final deployment
- **Domain setup**: Not required in config
- **Main domain**: Not configured

**Example**:

```bash
# Interactive: Select "Test deployment"
pnpm build:deploy:cf --both bayern
# Deploys to: sites/bayern/test/
# Accessible at: https://bayern.hero-pages.com
```

### ğŸš€ Final Deployment

- **Branch**: `main`
- **Access**: Via both `{productId}.hero-pages.com` AND main domain
- **Use case**: Production deployment
- **Domain setup**: Required in `multi-page-config.json`
- **Main domain**: Configured with www redirect and trailing slash normalization
- **DNS**: Creates DNS records for hero-pages.com subdomain
- **SSL**: Provisions SSL certificates automatically

**Example**:

```bash
# Interactive: Select "Final deployment"
pnpm build:deploy:cf --both bayern
# Deploys to: sites/bayern/main/
# Accessible at:
#   - https://bayern.hero-pages.com
#   - https://www.bayern.hero-pages.com (also works)
#   - https://www.example.com (from config)
#   - https://example.com (redirects from www)
```

**Note**: The www redirect goes from `www.{domain}` to `{domain}` (not the other way around). This is the standard practice for SEO and canonical URLs.

## Workflow Steps

### Full Pipeline (--both)

1. **Validation**
   - Validates Cloudflare environment variables
   - Checks site configuration

2. **Sitemap Generation**
   - Generates sitemaps for selected sites
   - Outputs to `public/sitemap-{productId}.xml`

3. **llm.txt Generation** (if `--generate-llm-txt`)
   - Generates llm.txt files using OpenAI API
   - Outputs to `public/llm-{productId}.txt`

4. **Static Build Generation**
   - Runs `generate-static.mjs` for each site
   - Outputs to `static-builds/{productId}/`

5. **R2 Deployment**
   - Uploads files to R2 bucket
   - Path: `sites/{siteId}/{branch}/`

6. **KV Route Setup**
   - Maps hostname to R2 path in KV store
   - Configures both hero-pages.com and main domain

7. **Worker Route Setup**
   - Ensures wildcard route `*.hero-pages.com/*`
   - Adds custom domain routes (final only)

8. **DNS Record Setup**
   - Creates CNAME records for `{productId}.hero-pages.com` (always)
   - Creates CNAME records for `www.{productId}.hero-pages.com` (always)
   - Adds custom domains to Worker for SSL certificate provisioning
   - Records are proxied through Cloudflare

9. **WWW Redirect Setup** (final only)
   - Creates 301 redirect: `www.{domain}` â†’ `{domain}` (preserves path and query)
   - Uses Zone-level Redirect Rules

10. **Trailing Slash Normalization** (final only)
    - Adds trailing slashes to paths that don't have one
    - Ignores files (paths containing ".")
    - Uses Zone-level Redirect Rules

## Troubleshooting

### Error: Cloudflare configuration is incomplete

**Problem**: Missing required environment variables

**Solution**:

1. Check `.env.local` file exists
2. Verify all required variables are set
3. Ensure no typos in variable names

### Error: No site path found for {productId}

**Problem**: Static build doesn't exist

**Solution**:

1. Run with `--generate` first to create builds
2. Or use `--both` to generate and deploy

### Error: Invalid Product ID

**Problem**: Product ID not in `multi-page-config.json`

**Solution**:

1. Check `src/data/multi-page-config.json`
2. Ensure site has `id` field
3. Use `--help` to see available products

### Warning: KV route setup failed

**Problem**: KV API access issue

**Solution**:

1. Verify `CLOUDFLARE_API_TOKEN` has KV permissions
2. Check `CLOUDFLARE_KV_NAMESPACE_ID` is correct
3. Verify KV namespace exists in Cloudflare Dashboard

### Warning: Could not ensure Worker route

**Problem**: Worker route configuration failed

**Solution**:

1. Verify `CLOUDFLARE_API_TOKEN` has Workers permissions
2. Check `CLOUDFLARE_WORKER_NAME` matches your worker
3. Ensure domain is added to Cloudflare zone
4. Manually add route in Cloudflare Dashboard if needed

### Warning: Could not create www redirect

**Problem**: Redirect rule creation failed

**Solution**:

1. Verify domain zone exists in Cloudflare
2. Check API token has Zone permissions
3. Manually create redirect in Cloudflare Dashboard:
   - Rules â†’ Redirect Rules â†’ Zone-level
   - Pattern: `www.{domain}/*`
   - Destination: `{domain}/$1`
   - Status: 301
   - Note: The redirect goes from www to non-www (not the other way around)

### Warning: Could not create trailing slash normalization rule

**Problem**: Trailing slash rule creation failed

**Solution**:

1. Verify domain zone exists in Cloudflare
2. Check API token has Zone permissions
3. Manually create rule in Cloudflare Dashboard:
   - Rules â†’ Redirect Rules â†’ Zone-level
   - Expression: `(not ends_with(http.request.uri.path, "/")) and not http.request.uri.path contains "."`
   - Action: Redirect
   - Destination: `concat(http.request.uri.path, "/")`
   - Status: 301
   - Preserve query string: Yes

### Warning: Could not create DNS records for hero-pages.com subdomain

**Problem**: DNS record creation failed

**Solution**:

1. Verify `hero-pages.com` zone exists in Cloudflare
2. Check API token has Zone permissions
3. Verify `CLOUDFLARE_WORKER_DNS_TARGET` is set correctly (or uses default)
4. Manually create DNS records in Cloudflare Dashboard:
   - DNS â†’ Records
   - Type: CNAME
   - Name: `{productId}.hero-pages.com`
   - Target: `{worker-name}.hero-pages.workers.dev` (or custom target)
   - Proxy status: Proxied
   - Repeat for `www.{productId}.hero-pages.com`

### Deployment succeeds but site not accessible

**Problem**: Routing not configured correctly

**Solution**:

1. Check KV route exists: Cloudflare Dashboard â†’ Workers & Pages â†’ KV
2. Verify Worker route exists: Workers & Pages â†’ {worker} â†’ Settings â†’ Triggers
3. Check DNS is configured correctly
4. Verify SSL/TLS mode is "Full" or "Full (strict)"

### llm.txt generation fails

**Problem**: OpenAI API issue

**Solution**:

1. Verify `OPENAI_API_KEY` is set and valid
2. Check API key has sufficient credits
3. Ensure API key has access to required models

## Best Practices

1. **Test First**: Always use test deployment before final

   ```bash
   pnpm build:deploy:cf --both bayern
   # Select "Test deployment"
   ```

2. **Verify Builds**: Check `static-builds/` before deploying

   ```bash
   ls -la static-builds/{productId}/
   ```

3. **Incremental Deploys**: Use `--deploy` for quick redeployments

   ```bash
   pnpm build:deploy:cf --deploy bayern
   ```

4. **llm.txt Timing**: Only generate llm.txt when content is finalized
   - Avoids unnecessary API costs
   - Ensures accurate content indexing

5. **Environment Security**: Never commit `.env.local` to git
   - Add to `.gitignore`
   - Use secure secret management in CI/CD

6. **Monitor Deployments**: Check Cloudflare Dashboard after deployment
   - Verify R2 bucket has files
   - Check KV routes are set
   - Confirm Worker routes are active
   - Verify DNS records are created
   - Check SSL certificates are provisioned (may take a few minutes)

7. **Use Post-Deploy Mode**: For routing fixes without redeployment

   ```bash
   pnpm build:deploy:cf --post-deploy bayern
   ```

   - Useful when you need to fix routes or redirects
   - No file upload required
   - Faster than full redeployment

8. **Deployment Type Prompting**: The script always prompts for deployment type
   - Even in command-line mode, you'll be asked to choose test or final
   - This prevents accidental production deployments

## Key Features

### Automatic DNS Management

The script automatically creates and manages DNS records for hero-pages.com subdomains:

- Creates CNAME records for `{productId}.hero-pages.com`
- Creates CNAME records for `www.{productId}.hero-pages.com`
- Adds custom domains to Worker for SSL certificate provisioning
- Records are proxied through Cloudflare (orange cloud)

### Static Build Path Detection

The script intelligently detects static builds in multiple locations:

- Checks `static-builds/{productId}/` first (primary)
- Falls back to alternative paths if primary doesn't exist
- Supports different build output configurations

### External API Support

For static builds, the script supports external API configuration:

- `NEXT_PUBLIC_API_URL`: External API endpoint for email, reCAPTCHA, etc.
- `NEXT_PUBLIC_API_TOKEN`: Bearer token for API authentication
- Script validates and reports API configuration status

### Text Editor Disabling

Cloudflare deployments automatically disable the text editor:

- Sets `ENABLE_TEXT_EDITOR=false` for static exports
- Prevents editor and SEO buttons from appearing in production
- Ensures clean static site experience

### Deployment Summaries

The script provides detailed summaries after deployment:

- **Deployment Summary**: Shows deployment status, R2 paths, accessible URLs
- **Post-Deployment Setup Summary**: Shows KV routes, DNS, redirects, trailing slash rules
- **Processing Summary**: Overall success/failure status

### Branch-Based Organization

Files are organized in R2 by branch:

- `sites/{siteId}/main/` - Production deployments
- `sites/{siteId}/test/` - Test deployments
- Branch is internal organization only (not in public URL)
